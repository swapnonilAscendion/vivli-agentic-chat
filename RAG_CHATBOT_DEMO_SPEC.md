# RAG CHATBOT DEMO - IMPLEMENTATION SPEC

## 🎯 Quick Overview (1-2 Days MVP)

A working RAG chatbot demo with:
- Single FastAPI endpoint: `POST /chat`
- Query → Embedding → Azure AI Search → LLM → Response
- Demo with 5-10 sample questions answered with sources

---

## 📋 IMPLEMENTATION PLAN

### PHASE 1: Setup (30 min)
- Create `rag-demo/` directory with basic structure
- Set up requirements.txt (FastAPI, Azure clients, etc.)
- Configure environment variables (.env)
- Verify Azure AI Search index exists

### PHASE 2: Core Components (60 min)
- **embeddings.py** - Wrapper for Azure OpenAI embeddings
- **retrieval.py** - Query vector DB, get top-K docs
- **llm.py** - Call Azure OpenAI LLM with context
- **models.py** - Pydantic request/response models
- **config.py** - Environment & settings

### PHASE 3: API Endpoint (45 min)
- **main.py** - Single `/chat` endpoint that orchestrates flow

### PHASE 4: Testing (30-45 min)
- Unit tests for each component
- Integration test for full flow
- Sample test queries

## 🔑 Key Prompts for Code Generation

### SYSTEM PROMPT (for developers)
**Purpose:** Guide implementation of the chatbot demo

You are building a RAG (Retrieval-Augmented Generation) chatbot demo for Vivli in 1-2 days. Focus on:

1. **Intent Classification**: Classify query as FAQ, Data Request Related, or Escalation
2. **Knowledge Base Retrieval**: Use Azure AI Search to find relevant docs
3. **LLM Response Generation**: Use Azure OpenAI (GPT-4o-mini) to generate natural language responses
4. **Response Formatting**: Format responses according to Vivli standards
5. **Error Handling**: Escalate when confidence is low or docs not found

**Non-functional Requirements:**
- Latency: < 500ms per request
- Confidence threshold: >= 0.6 for automated responses
- No hallucinations: Response must be grounded in retrieved docs
- Always include disclaimer about AI limitations

---

### INTENT CLASSIFICATION PROMPT (system message for LLM)

You are an intent classifier for a research platform. Classify the user's query into ONE of these categories:

**FAQ**: General questions about the platform, process, policies, procedures
- Keywords: "how", "what", "where", "when", "process", "policy", "requirements", "timeline"
- Examples:
  - "How do I submit a data request?"
  - "What documents do I need?"
  - "What is the process timeline?"

**DATA_REQUEST_RELATED**: Questions about a specific data request's status or feedback
- Keywords: "status", "my request", "update", "feedback", "revision", "reviewed"
- Examples:
  - "What is the status of my request #12345?"
  - "Have you reviewed my request?"
  - "Can I check when my submission will be reviewed?"

**ESCALATION**: Requests for human help or escalation
- Keywords: "help", "escalate", "human", "staff", "not helpful", "need assistance"
- Examples:
  - "I need help"
  - "Escalate this to a human"
  - "This isn't helping"

**HYBRID**: Multiple questions that fit different categories
- Example: "How do I submit a data request? [FAQ] And what's the status of my existing request? [DATA_REQUEST]"

Return JSON with:
```json
{
  "intent": "FAQ|DATA_REQUEST_RELATED|ESCALATION|HYBRID|UNKNOWN",
  "confidence": 0.0-1.0,
  "keywords_detected": ["list", "of", "keywords"]
}
```

---

## 📊 Intent Classification Decision Tree

```
User Query
    ↓
Extract keywords & calculate semantic similarity
    ↓
Score: FAQ, Data Request, Escalation
    ↓
Determine highest score(s):
    ├─ Only FAQ > 0.7 → FAQ
    ├─ Only Data Request > 0.7 → DATA_REQUEST_RELATED
    ├─ Only Escalation > 0.7 → ESCALATION
    ├─ Multiple scores > 0.5 → HYBRID
    └─ No score > 0.7 → UNKNOWN (escalate)
    ↓
Route to appropriate handler:
    ├─ FAQ → Retrieve docs + Generate with FAQ prompt
    ├─ DATA_REQUEST_RELATED → Fetch status + Generate with status prompt
    ├─ HYBRID → Fetch status THEN retrieve docs + Combined response
    ├─ ESCALATION → Send to human
    └─ UNKNOWN → Send to human
```
